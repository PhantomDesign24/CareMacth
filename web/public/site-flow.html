3. 확장 시나리오

3-1. 간병 연장 (Extension)

보호자가 추가 일수 신청 → 간병인 수락 → 보호자 결제 → 적용

1보호자: 연장 신청
기존 간병인 / 새 간병인 선택. 기존 간병인은 수락 단계 필요.
POST /api/contracts/:id/extend PENDING_CAREGIVER_APPROVAL
↓
2간병인: 수락 / 거절 (24시간 내)
미응답 시 자동 EXPIRED, 수락 시 보호자에게 결제 안내
POST /api/contracts/:cid/extension/:eid/approve POST /api/contracts/:cid/extension/:eid/reject PENDING_PAYMENT 전이
↓
3보호자: 결제 (1시간 내)
미결제 시 자동 EXPIRED (cron 5분 주기)
POST /api/payments (body: { contractId, method, extensionId }) CONFIRMED → endDate 갱신
변수 발생 시 문제점
  • 간병인 무응답: 24h cron 으로 EXPIRED 처리 + 보호자 알림. 다음 신청 가능.
  • 중복 신청: 트랜지션 락으로 in-flight 가드. 진행 중 연장 있으면 400.
  • 동시 수락 race: updateMany where status=PENDING_CAREGIVER_APPROVAL 로 1회만 통과.
  • 결제 미완료 만료: 1h cron → EXPIRED. 보호자 재시도 시 새 연장 신청 필요.
  • EXTENDED 상태에서 다시 연장: 현재 ACTIVE 만 허용 → 400.

3-2. 추가비 청구 (AdditionalFee)

간병인이 야간/위험수당 청구 → 보호자 승인 → 합계에 가산

1간병인: 추가비 요청 POST /api/payments/additional-fees
↓
2보호자: 승인 / 거절 POST /api/payments/additional-fees/:id/approve POST /api/payments/additional-fees/:id/reject 트랜지션 락: 미처리 상태에서만 1회
↓
3정산에 자동 합산
간병인 Earning 에 반영, 보호자 결제 내역에 표기
변수 발생 시 문제점
  • 중복 승인: 락으로 차단 (이전: 알림 두 번 발송).
  • 승인 후 거절 시도: 409 응답.
  • 금액 음수: 입력 검증으로 400.

3-3. 환불 (Refund)

2단계 흐름 — 보호자 요청 → 관리자 검토 → 환불 실행

1보호자: 환불 요청
사유 + 금액(전체/부분) 입력
POST /api/payments/:id/refund refundRequestStatus = PENDING
↓
2관리자: 검토 + 승인 / 거절
승인 시 토스 cancel API 호출, Earning 자동 차감, 전액 환불 시 계약 자동 취소
POST /api/admin/payments/:id/refund-approve POST /api/admin/payments/:id/refund-reject refundReviewedAt 클레임 락
↓
3정산 / 알림
미정산 Earning 차감, 보호자/간병인 양측 푸시
PARTIAL_REFUND / REFUNDED
변수 발생 시 문제점
  • 동시 관리자 승인 race: 실손실 이전: Earning 두 번 차감 (간병인 정산 손실). 수정: refundReviewedAt 클레임 락.
  • 토스 API 실패: 락 해제(refundReviewedAt=null) 후 재시도 가능.
  • 환불 금액 초과: 결제액 초과 입력 시 400.
  • 이미 PARTIAL_REFUND 인 결제 재요청: 가능 (남은 금액 한도 내).

3-4. 분쟁 (Dispute)

보호자/간병인이 분쟁 신고 → 관리자 처리 → 정산 보류 또는 환불 연계

1분쟁 신고
카테고리(서비스 품질/금액/사고 등), 증거 첨부
POST /api/disputes
↓
2관리자: 처리
PROCESSING → 양측 의견 수렴 → 환불 / 패널티 / 계약 취소 등 결정
PATCH /api/disputes/admin/:id
↓
3RESOLVED
계약 취소 시 관련 분쟁 자동 RESOLVED

4. 위험 변수 — Race Condition / 캐시 우회

4-1. Race Condition (동시 요청)

두 요청이 동시 도착할 때 발생하는 문제. 모두 트랜지션 락 패턴 적용 완료.

지점위험적용한 락
signContract 같은 사람이 동시 서명 시도 → 양측 통과 updateMany where signedAt=null
extendContract 동시 신청 시 in-flight 가드 통과 → 중복 생성 SELECT FOR UPDATE + 트랜잭션 내 검증
approveExtension 동시 수락 시 PENDING_PAYMENT 두 번 전이 updateMany where status=PENDING_CAREGIVER_APPROVAL
cancelContract 실손실 보호자+간병인 동시 취소 → 환불 두 번 처리 updateMany where status=ACTIVE
createContract 같은 간병인이 두 careRequest 에 동시 매칭 SELECT FOR UPDATE + 실제 contract 카운트
confirmPayment 실손실 동시 confirm → ESCROW + FAILED 충돌, totalAmount 이중 가산 tossPaymentKey 클레임 패턴
createPayment 동시 DIRECT 결제 → Earning 두 건 생성 (이중 정산) SELECT FOR UPDATE Contract + 트랜잭션 내 dedup (PENDING/COMPLETED/ESCROW)
approveRefundRequest 실손실 동시 관리자 승인 → Earning 두 번 차감 refundReviewedAt 클레임 패턴
approveAdditionalFee / reject 중복 승인 → 알림 2회 발송 updateMany where approved=false AND rejected=false

4-2. 캐시 필드 우회 (단일 진실 원천 위반)

denormalized 캐시 필드가 실제 데이터와 동기화 안 될 때 발생.

필드이전 검사현재 검사
caregiver.workStatus 이 필드만 보고 진행 중 판정 → 임의로 AVAILABLE 변경 시 우회 실제 Contract 테이블에서 ACTIVE/EXTENDED/PENDING_SIGNATURE 카운트로 판정
실제 검증 결과 
대상: cg2@test.com (ACTIVE 계약 보유)
workStatus + 인증서류 모두 통과 상태로 강제 설정
지원 시도 → ✅ 차단: 400 "현재 진행 중인 계약이 있어..."

4-3. 외부 API 실패 처리

  • 토스 결제 confirm 실패: tossPaymentKey 클레임 후 Toss 호출 실패 시 PENDING 상태에서만 FAILED 마킹 (이미 ESCROW 됐으면 overwrite 금지).
  • 토스 cancel 실패 (환불): 락 해제(refundReviewedAt=null) → 재시도 가능.
  • FCM 푸시 실패: 알림 DB 저장은 성공, 푸시만 pushSuccess=false. 토큰 만료 시 자동 제거.
  • Firebase 미초기화: 푸시 발송만 건너뛰고 비즈니스 로직은 정상 동작.

6. 시나리오 테스트 상세

test-extension-flow.ts (18건)

  • 서명: 단독, 양측 완료, 중복, 크기 초과
  • 연장: 신청, 수락, 거절, 권한, 자동 만료(24h/1h), 신규 간병인
  • 동시성: 서명, 연장 신청, 수락, 결제 race

test-advanced-scenarios.ts (21건)

  • 고위험: 환불 race, 같은 간병인 동시 매칭
  • 입력 검증: 음수 포인트, 보유 초과, 입력 형식
  • 흐름: DIRECT 결제 + 토스 confirm 충돌, EXTENDED 다중 연장
  • 권한: 다른 보호자 계약 접근/취소/PDF 차단
  • SUSPENDED: 로그인 + 지원 차단
  • PDF: ACTIVE / CANCELLED 워터마크

test-domain-scenarios.ts (18건)

  • 환불: 요청, 동시 승인 race, 거절
  • 추가비: 요청, 동시 승인 race, 검증
  • 지원: 중복 차단, 매칭 시 자동 처리
  • 리뷰: 작성, 중복 차단, 평점 갱신, 권한
  • 패널티: 누적 → SUSPENDED 자동 전환
  • Earning: DIRECT 동시 결제 race
